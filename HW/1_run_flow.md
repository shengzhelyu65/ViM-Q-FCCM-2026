## Prerequisites

Before running the flow, ensure:

1. **Environment Setup**: Source environment variables
   ```bash
    cd HW
   source setup_env.sh
   ```

2. **Test Data**: Verify data directories exist
    ```bash
    ls data/
    # Should show: bin_float32_block/, image_float32_block/, ref_float32_block/, ssm_float32/
    ```

   **Note**: The `ssm_float32/` folder contains SSM test data and can be generated by:
    ```bash
    python3 util_generate_ssm_data.py
    # This generates: u, delta, z_silu, B, C, A, D, and reference output files
    ```

## Complete Development Flow

The ViM-Q hardware development consists of 11 steps:

### Step 1: HLS C-Simulation

Verify HLS C++ code correctness before synthesis.

```bash
# Run C-simulation for all modules
python3 step1_hls_sim.py

# Or run for specific module
python3 step1_hls_sim.py LINEAR_BLOCK
python3 step1_hls_sim.py EMBED
```

**What happens:**
- Creates HLS project directories under `instances/proj_{MODULE}/`
- Runs Vitis HLS C-simulation
- Generates logs in `logs/hls_run_summary_*.log`

---

### Step 2: HLS C-Synthesis

Synthesize HLS C++ code to Verilog RTL.

```bash
# Run C-synthesis for all modules
python3 step2_hls_syn.py

# Or run for specific module
python3 step2_hls_syn.py LINEAR_BLOCK
python3 step2_hls_syn.py EMBED
```

**What happens:**
- Generates Verilog RTL from C++ code
- Creates resource utilization reports
- Generates timing reports
- Generates log files in `logs/hls_run_summary_*.log`

---

### Step 3: HLS C-Cosimulation

Verify synthesized Verilog against C++ reference model.

```bash
# Run C-cosimulation for all modules
python3 step3_hls_cosim.py

# Or run for specific module
python3 step3_hls_cosim.py LINEAR_BLOCK
python3 step3_hls_cosim.py EMBED
```

**What happens:**
- Runs RTL cosimulation using synthesized Verilog
- Compares cosimulation results with C++ model
- Generates cosimulation reports
- Generates log files in `logs/hls_run_summary_*.log`

---

### Step 4: Print HLS Resource Summary

Collect and print consolidated resource/latency summaries after synthesis.

```bash
python3 step4_print_resource.py
```

**What happens:**
- Parses per-module HLS report outputs
- Prints a compact summary for quick comparison across modules

---

### Step 5: Copy Verilog to SpinalHDL

Prepare Verilog files for SpinalHDL integration.

```bash
python3 step5_spinal_flow.py
```

**What happens:**
- Copies Verilog from HLS synthesis
- Combines all .v files into `SPINAL/src/main/verilog/{MODULE}/all.v`
- Adds Verilator lint directives
- Copies .dat files (for ROM initialization)
- Fixes readmemh paths for Verilator

---

### Step 6: Compile SpinalHDL Code

Build the SpinalHDL simulation environment.

```bash
cd SPINAL
sbt compile
```

**What happens:**
- Compiles all Scala files in `src/main/scala/`
- Generates Scala classes in `target/scala-2.11/classes/`

---

### Step 7: Run Individual Module Simulations

Test each module independently using Verilator.

```bash
cd SPINAL

# Test EMBED module
sbt "runMain simulate_embed"

# Test CONV module
sbt "runMain simulate_conv"

# Test LINEAR_BLOCK module
sbt "runMain simulate_linear_block"

# Test NORM_SUM module
sbt "runMain simulate_norm_sum"

# Test SSM module
sbt "runMain simulate_ssm"

# Test SMOOTH module
sbt "runMain simulate_smooth"

# Test PATCH_OPS module
sbt "runMain simulate_patch_ops"
```

**Each simulation does:**
1. Loads HLS-generated Verilog as BlackBox
2. Initializes AXI memory simulators
3. Loads test data from `data/` directory
4. Configures module via AXI-Lite control interface
5. Runs simulation with Verilator
6. Reads output from AXI memory
7. Compares with reference from `data/ref_float32_block/`
8. Reports MSE, MAE, RMSE, and max error

**Expected output:**
```
[info] === EMBED Simulation Test ===
[info] D_MODEL: 192, NUM_PATCHES: 197
[info] INPUT: 224x224x3
[info] Loading image data...
[info] Computation completed in 539400 cycles
[info] Comparing results...
[info] EMBED Output: MSE=0.000002, MAE=0.001479, RMSE=0.001548
[info] ✓ EMBED Output: PASSED
```

---

### Step 8: Run Full ViM Pipeline Simulation

Test complete ViM pipeline with all modules connected.

```bash
cd SPINAL
sbt "runMain simulate_vim_full"
```

**What happens:**
- Instantiates all modules (EMBED, NORM_SUM, LINEAR, CONV, SMOOTH, SSM, PATCH_OPS)
- Connects modules via DaisyChain
- Runs single-layer pipeline simulation
- Tracks latency for each module
- Tests complete data path from image embedding through SSM
- Provides cycle-accurate end-to-end latency

After running `sbt "runMain simulate_vim_full"` in `HW/SPINAL`, detailed cycle-accurate latency data for default `224x224` input resolution are written to:
- `HW/SPINAL/latency_report.log`
- `HW/SPINAL/latency_report.csv`

---

### Step 9: Generate ViM Accelerator Verilog

Generate top-level ViM accelerator RTL for Vivado integration.

```bash
cd SPINAL
sbt "runMain generate_vim_accelerator"
```

**What happens:**
- Generates `ViM_ACCELERATOR.v` (top-level Verilog)
- Exposes all AXI4 memory interfaces
- Exposes all AXI4Lite control interfaces
- Includes DaisyChain control signals

---

### Step 10: Create Vivado Block Design

Create Xilinx IP catalog entry for ViM accelerator.

```bash
python3 run_vivado_flow.py
```

**What happens:**
1. Packages ViM accelerator as Xilinx IP
2. Creates Vivado block design project
3. Instantiates ViM accelerator IP
4. Configures AXI4 interconnect for DDR
5. Exposes all control interfaces via AXI-Lite

---

### Step 11: Run Vivado Implementation

Generate FPGA bitstream.

```bash
python3 run_vivado_implementation.py
```

**What happens:**
1. Synthesizes design
2. Places and routes design
3. Creates bitstream file
4. Exports post-implementation reports to `vivado_reports/post_impl/`:
   - `utilization_post_impl.rpt` (area/utilization summary)
   - `utilization_hier_post_impl.rpt` (hierarchical utilization)
   - `power_post_impl.rpt` (power)

If you only want to regenerate post-implementation utilization/power reports (without rerunning implementation), run:

```bash
python3 run_vivado_implementation.py --reports-only
```
